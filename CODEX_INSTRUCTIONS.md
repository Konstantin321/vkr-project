# Инструкции для Codex по проекту ВКР

Перед любыми изменениями обязательно прочитай VKR_CONTEXT.md.

Проект: веб-система формирования фонда оценочных средств по дисциплинам базовой кафедры.

Стек:
- PHP
- PostgreSQL
- HTML/CSS
- Bootstrap/MDB
- MVC

Главное правило:
не переписывать проект с нуля, а вносить точечные изменения в существующие файлы.

Единый стиль интерфейса:
- для новых и обновляемых публичных страниц использовать общий layout:
  - `public/includes/layout_start.php`;
  - `public/includes/layout_end.php`;
- общие стили хранить и развивать в `public/assets/css/app.css`;
- сохранять визуальную стилистику проекта:
  - тёмное левое меню;
  - светлый серо-голубой фон контента;
  - белые карточки;
  - синие акценты;
  - аккуратные таблицы;
  - плавные hover-состояния кнопок и пунктов меню;
- не возвращать inline CSS, кроме случаев, когда он нужен существующему JavaScript для динамического `display`.

Авторизация:
- рабочие публичные страницы должны вызывать `Auth::requireAuth()` до выполнения бизнес-логики;
- текущего пользователя брать через `Auth::id()` / `Auth::user()`, не использовать заглушки вида `user_id = 1`;
- локальная авторизация реализована как адаптер в `app/auth/Auth.php`, чтобы позже можно было подключить кафедральный SSO/LDAP без переписывания страниц;
- изменения схемы для локального входа описаны в `storage/auth_schema.sql`.

Если меняешь код:
1. сначала объясни, какие файлы будут изменены;
2. меняй только нужные участки;
3. не удаляй уже работающую логику;
4. сохраняй MVC-структуру;
5. учитывай текущую БД PostgreSQL;
6. после изменения кратко напиши, что проверить.

Важная логика:
- преподаватель работает с заданиями и наборами;
- администратор только управляет правами;
- обучающийся проходит контроль;
- задание может входить в несколько наборов через task_set_items;
- нельзя ломать уже реализованные модули заданий, наборов, попыток, ответов и проверки.
